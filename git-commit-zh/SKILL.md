# 极简 Git Commit 规范指南

好的 commit 规范能让代码历史清晰可读。业界最流行的是 Conventional Commits（如 `feat:`、`fix:`）和 gitmoji，但在深度依赖任务管理系统（如 Jira、Linear）的现代团队中，这些传统规范往往显得冗余且宽泛。

本文推荐一套**基于任务 ID 的极简符号系统**，并简要分析传统方案的局限。

## 推荐方案：基于任务 ID 的符号系统

在传统的规范中，`feat` 这个词过于宽泛，导致 commit 历史里满屏都是 `feat:`，完全看不出具体做了什么。

为了解决这个问题，我们推荐放弃冗余的英文单词前缀，直接用**括号类型**和**首字母**来区分任务性质，并强制绑定**任务 ID**。

这套系统将改动分为三大类：

### 1. 影响功能的改动（方括号 `[]`）

功能性的改动对外部用户可见，是最核心的提交。使用方括号表示强调。

- `[T001]`：**Task**，代表新功能、需求变更。
- `[B001]`：**Bug**，代表缺陷修复。

### 2. 代码质量的改动（圆括号 `()`）

这类改动不改变外部行为，主要为了提升代码内部的健康度。

- `(R001)`：**Refactor**，代表重构、性能优化等代码调整。

### 3. 辅助性的改动（大括号 `{}`）

不涉及核心业务逻辑的边缘改动。

- `{D001}`：**Doc**，代表文档修改。
- `{C001}`：**Chore**，代表构建脚本、CI 配置等杂项。

### 进阶：使用冒号 `:` 标记影响范围（Scope）

为了在 monorepo 或大型项目中更精准地定位改动，我们可以在任务 ID 后面加上冒号 `:` 和受影响的模块（scope）。

格式为：`[ID:scope]`。如果有多个平级的模块受影响，可以继续用冒号连接：`[ID:scope1:scope2]`。使用冒号（命名空间语法）不仅视觉上轻量优雅，而且非常利于 `git log --grep=":android"` 的精准检索。

### 实际示例

*注意：所有可用的任务 ID 均应在项目根目录的 `TASKS.md` 文件中统一管理和追踪。*

```text
[T123] add OAuth login support
[T001:server] implement payment webhook
[B002:server:android] fix session timeout issue across platforms
(R089:core) extract payment webhook processor
{D012} update API documentation
{C013:web} upgrade webpack to v5
```

### 这套系统的核心优势

1. **消灭冗余前缀**：不再需要 `feat(server): [T123] add login`，括号形状、首字母和冒号本身就是语义，极其克制。
2. **强制关联任务**：commit 必须带 ID，天然与任务管理系统绑定。回溯问题时，直接拿 ID 去需求文档里查，而不是在茫茫的 `feat:` 历史中猜背景。
3. **视觉高度清晰**：`[]`、`()`、`{}` 的视觉区隔非常强。扫一眼 commit log，立刻能区分哪些是新功能（方括号），哪些是代码重构（圆括号）。
4. **范围检索极简**：引入 `:scope` 后，由于冒号连接紧密且天然防冲突，开发者用 `git log --grep=":android"` 就能百分百精准找出一个模块的所有改动。

## 传统方案的局限性

为什么我们要放弃主流的规范？因为它们在实际工程中都暴露出了一些水土不服的问题。

### Conventional Commits 的问题

Conventional Commits 规定了 `<type>: <description>` 的格式。

- **`feat` 过于宽泛**：写一个按钮是 `feat`，重构整个底层也是 `feat`。当历史记录里全是 `feat:` 时，这个前缀就失去了信息量。
- **Type 膨胀**：团队为了区分不同改动，往往会定义十几个 type（`perf`, `style`, `test` 等），增加了不必要的记忆负担。

### gitmoji 的局限

gitmoji 提倡用 emoji 替代文字前缀（如 🐛 代表 fix，✨ 代表 feat）。

- **选择过载**：官方定义了 140+ 种 emoji，大家往往凭感觉选。有人用 🐛 有人用 🔧 都表示修复，反而导致规则混乱。
- **跨平台兼容差**：在不同的操作系统、IDE、甚至终端里，emoji 的显示效果可能完全不同，有些环境甚至会显示乱码。
- **过度娱乐化**：有些团队追求"每条 commit 都带 emoji"，改个错别字也要配个 ✏️，形式主义分散了对代码本身改动的注意力。

如果团队确实喜欢视觉化，gitmoji 最好仅作为补充，而不是核心规范（比如限制在 5 个常用 emoji 以内）。

## 规范落地的关键

再好的规范，执行不下去等于没有。

1. **Commit 必须原子化**：同一个功能或任务相关的修改，应该打包在一个 commit 中提交。关于粒度的把控，业界虽然有“全量打包”和“原子化拆分”的争论，但**我们要求必须原子化拆分**：
   - 比如实现一个功能时，把“重构旧代码”作为一个 `(R001:xxx)` 提交，把“新增功能逻辑”作为一个 `[T001:xxx]` 提交。
   - **原因**：将不改变外部行为的重构（Refactor）与影响功能的新增（Task）混在一个大 commit 里，是引发隐藏 Bug 且极难排查的元凶。严格按改动性质拆分，历史更清晰，在使用 `git bisect` 二分查找时能更精准地定位到具体的变动。
2. **CI 强制检查**：通过 git hooks 或 CI 管道，使用正则严格检查。人工 review commit message 是浪费时间，应交给脚本自动拦截。

   **`commit-msg` hook 校验脚本参考**：
   ```bash
   #!/usr/bin/env bash
   COMMIT_MSG=$(cat $1)
   # 支持 [T123] 或 [T123:server:android] 的正则
   PATTERN="^(\[([TB])[0-9]+(:[a-zA-Z0-9,-]+)*\]|\([R][0-9]+(:[a-zA-Z0-9,-]+)*\)|\{[DC][0-9]+(:[a-zA-Z0-9,-]+)*\}) .+"

   if [[ ! $COMMIT_MSG =~ $PATTERN ]]; then
     echo "Error: 提交格式不规范！"
     echo "合法示例: [T123:server:android] add feature"
     exit 1
   fi
   ```
3. **处理无 ID 场景**：对于极少数确实没有任务 ID 的微小改动，可以约定使用特定的 fallback 标记，如 `{chore}` 或 `{typo}`。
4. **从新项目开始**：历史包袱不要回头改，从下一个新项目或新版本开始强制推行新规范。

## 总结

规范是为了让人看懂历史，不是为了展示团队有多专业。基于任务 ID 的符号系统放弃了表面的形式感，直击工程痛点：**让每一行 commit 都能溯源到具体的需求，并且一目了然。**
