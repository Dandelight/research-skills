# Task Decomposition Skill - 任务分解与编排

## 一、触发识别（自动检测）

**强制触发条件（满足任一）：**
- 用户输入包含："分解"、"拆分"、"拆解"、"执行计划"、"子任务"、"里程碑"、"依赖关系"、"并行执行"、"工作流编排"
- 用户要求将复杂项目/需求/目标转化为可执行步骤
- 用户需要协调多个Agent或团队成员工作
- 任务涉及多阶段、多组件或多技术栈

**禁止触发条件（满足任一）：**
- 单步骤明确任务："翻译这段话"、"修复这个bug"、"解释这个概念"
- 用户明确表示："直接做"、"不用分解"、"现在就执行"
- 任务预计完成时间 < 3分钟

## 二、分解前门禁（CRITICAL - 不满足则停止）

在生成任何子任务前，**必须**完成以下验证：

### 2.1 信息密度评分（5要素检查）
用 **0-5分制** 评估任务信息充分度：

| 要素       | 检查项                                           |
| ---------- | ------------------------------------------------ |
| 🎯 目标     | 具体而非模糊（"实现OAuth2登录"✅ vs "改进认证"❌） |
| 📦 范围     | 明确的 In-scope / Out-of-scope 边界              |
| 👤 上下文   | 用户画像（who）或技术栈/约束（what）             |
| ✅ 成功标准 | 可衡量的验收标准（3-5条）                        |
| ⚠️ 约束     | 依赖、时间、预算、法规限制                       |

**评分与行动：**
- **0-2分（INSUFFICIENT）**：**立即停止**，询问3-5个具体问题（见2.2节）
- **3-4分（MARGINAL）**：继续分解但**记录所有假设**，并在输出中标注风险
- **5分（SUFFICIENT）**：直接进入分解

### 2.2 强制询问模板（信息不足时）

**禁止**：在信息不足时直接猜测分解。
**必须**：使用以下格式询问，一次只问最紧急的1-3个问题：

```markdown
⚠️ **信息不足，无法安全分解**

当前任务信息密度：[X]/5 分
缺失关键要素：[列举]

为了创建可执行的子任务，我需要澄清：

1. **[具体问题]**：（如：性能优化的具体指标是什么？）
2. **[具体问题]**：（如：涉及哪些用户角色？）
3. **[具体问题]**：（如：是否有技术栈限制？）

**请提供以上信息，或确认我可以基于以下假设继续**：
- 假设1：[具体假设]
- 假设2：[具体假设]
```

## 三、核心分解方法论（Layering）

### 3.1 活动驱动分解（Activity-Based）

**永远**按"**做什么**"（Activity）分解，而非"谁做"（Role）。

✅ **正确**："设计数据库Schema" → "实现API端点" → "编写单元测试"
❌ **错误**："后端工程师做X" → "前端工程师做Y"

### 3.2 依赖类型标记（强制使用）

为每个子任务标记依赖关系：

| 符号   | 类型 | 含义                       | 示例                       |
| ------ | ---- | -------------------------- | -------------------------- |
| `->`   | 顺序 | B需要A的输出               | `设计Schema -> 实现API`    |
| `\|\|` | 并行 | 可同时执行，无共享依赖     | `竞品A分析 \|\| 竞品B分析` |
| `?`    | 可选 | 非核心，有则更好           | `性能优化?`                |
| `R`    | 资源 | 需要共享资源（顺序或池化） | `R[数据库迁移]`            |

### 3.3 执行策略决策树

```markdown
if (所有检查通过):
    ✅ **并行执行**（Parallel）
else:
    📝 **顺序执行**（Sequential）

检查清单：
- [ ] 无输出依赖（B不需要A的结果）
- [ ] 无共享状态修改（或只读）
- [ ] 可独立验证
- [ ] 创建文件时路径唯一（无冲突）
- [ ] 无资源竞争
```

**混合策略模式（推荐）：**
```markdown
Group 1 (并行): 需求分析 || 竞品调研 || 技术选型
       ↓
顺序: 架构设计（需要Group 1结果）
       ↓
Group 2 (并行): 后端API || 前端UI || 测试用例
       ↓
顺序: 集成测试 -> 部署上线
```

## 四、子任务质量标准（5大必须）

每个生成的子任务**必须**满足：

1. **原子性（Atomic）**：只做一件事，无内部分支决策
2. **可验证（Verifiable）**：完成后能明确说"Done"
3. **独立性（Independent）**：能独立开发（或依赖已明确标注）
4. **尺寸合适（Sized）**：1-2天完成，1-2 story points，不超过4小时纯工作
5. **有价值（Valuable）**：交付用户价值或技术价值，非"为了拆分而拆分"
6. **有前后依赖**：每个子任务都有明确的依赖前序和后序任务关系，不能独立完成，起始任务可以没有前序任务，终止任务可以没有后序任务。

## 五、Agent编排模板（FOCUS/EXCLUDE法）

如需将子任务委托给Agent执行，**必须**使用以下结构化提示词：

```markdown
FOCUS: [完整具体的任务描述]
    - 具体要做什么（动词+名词+范围）
    - 关键参数/配置（如：bcrypt cost factor 12）
    - 包含的技术细节

EXCLUDE: [明确边界 - 防止范围蔓延]
    - 不包含的功能A（说明原因）
    - 不修改的模块B
    - 不实现的相关功能（标记为后续任务）

CONTEXT: [背景与约束]
    - 技术栈: [具体栈]
    - 遵循模式: [现有代码/文档路径]
    - 前置结果: [上一步的输出摘要]
    - 特殊约束: [安全/性能/法规要求]

OUTPUT: [预期交付物 - 确切路径]
    - 文件1: [路径]
    - 文件2: [路径]
    - 结构化结果:
        - 创建/修改: [文件列表]
        - 摘要: [1-2句话]
        - 测试: [状态]
        - 阻塞: [如有]

SUCCESS: [可衡量的完成标准]
    - 标准1（可测试）
    - 标准2（可验证）
    - 标准3（符合模式）

TERMINATION: [停止条件]
    - 完成成功
    - 被[具体阻塞]阻塞
    - 达到最大3次重试
```

**创建文件时的额外要求（DISCOVERY_FIRST）：**
```markdown
DISCOVERY_FIRST: 开始前先探索环境：
    - 检查现有模式：[具体命令]
    - 识别框架：[查找测试框架/ORM等]
    - 确认命名约定：[文件/变量命名模式]
    - 确定输出路径：[基于发现的唯一路径]
```

```markdown
3. 成功指标是什么？（转化率提升X%，页面加载<2秒，NPS提升？）
4. 范围边界？（仅移动端？仅结账流程？排除后台管理？）

**请回答以上问题**，我将为您创建精准的分解计划。
```

## 九、故障恢复（Error Handling）

### 9.1 连续失败处理
- 如**连续2个子任务**执行失败，**立即停止**并询问："2步失败，是否复盘计划？"
- 最多**3次重试**，之后必须提交给用户决策

### 9.2 重试策略模板
```markdown
🔄 **重试策略**（第[N]/3次）

**失败原因**: [诊断]
**策略调整**:
- FOCUS细化: [更具体的描述]
- EXCLUDE扩展: [新增排除项]
- CONTEXT增强: [更多约束/示例]
- OUTPUT明确: [确切文件路径]

**模板优化**: [具体修改内容]
```

### 9.3 部分成功处理
当Agent交付部分结果：
- 评估：**可独立交付？** → 接受并创建新任务补全
- 评估：**不完整无用？** → 整体重试
- 决策：**接受部分 + 新Agent补全缺失**

## 十、快速参考（Quick Reference）

| 场景                | 立即行动                               |
| ------------------- | -------------------------------------- |
| 用户说"分解这个"    | 检查信息密度 -> 评分<3则询问，≥3则分解 |
| 任务含"和"/"以及"   | 拆分为并行组（检查真无依赖）           |
| 任务含"然后"/"之后" | 标记为顺序依赖（->）                   |
| 涉及文件创建        | 确保路径唯一，使用DISCOVERY_FIRST      |
| 涉及外部API         | 创建 docs/interfaces/[service].md      |
| 信息模糊            | **停止并询问**，禁止猜测               |
| 用户说"直接做"      | 跳过分解，进入执行模式                 |

**最终检查（Final Gate）**：
输出前确认最后一句是**建设性结论**或**可执行的第一步建议**（禁止以开放式问题或纯风险描述结尾）。
